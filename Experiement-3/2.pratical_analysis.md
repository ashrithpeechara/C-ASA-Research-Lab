# Practical Analysis & Solution

# Overview

The objective of this experiment was to implement **dynamic routing** using the **Open Shortest Path First (OSPF)** protocol with **FRRouting (FRR)** inside Docker containers. Unlike static routing, OSPF enables routers to automatically discover neighboring routers, exchange routing information, and maintain updated routing tables without manual intervention. This experiment establishes the dynamic networking foundation required for the later implementation of Autonomic Networking and the proposed Cognitive Autonomic Service Agent (C-ASA).

---

# Step 1: Docker-Based Router Deployment

Three virtual routers (Router1, Router2, and Router3) were deployed using Docker Compose. Each router was created as an independent Docker container and connected through a common bridge network. The `docker-compose.yml` file automated the deployment process by assigning hostnames, IP addresses, network interfaces, and mounting the FRRouting configuration files.

### Practical Analysis

Using Docker eliminates the need for physical networking hardware while providing an isolated and reproducible environment. Each container behaves like an independent Linux router, making it possible to build and test enterprise-level routing protocols in a lightweight virtual infrastructure.

### Solution

The routers were successfully deployed and connected within the Docker bridge network, providing the virtual infrastructure required for dynamic routing.

# Step 2: FRRouting Configuration

Each router was configured using two configuration files:

- **daemons** – Enables the required routing services (Zebra, OSPF, and Static Routing).
- **frr.conf** – Defines the router hostname, OSPF process, and directly connected networks to be advertised.

Although all routers used the same daemon configuration, each router advertised different networks based on its connected interfaces.

### Practical Analysis

The `daemons` file determines which routing services are started when FRRouting initializes. Enabling only the required services minimizes unnecessary resource usage. The `frr.conf` file provides router-specific configurations, allowing each router to participate in OSPF while advertising its own connected networks.

### Solution

FRRouting was successfully initialized on all three routers, and each router was configured to participate in OSPF Area 0.

# Step 3: OSPF Neighbor Discovery

After starting the routers, the OSPF daemon automatically exchanged Hello packets between neighboring routers. This process established OSPF adjacencies, allowing routers to identify and communicate with one another.

The neighbor relationships were verified using:

```bash
show ip ospf neighbor
```

### Practical Analysis

Neighbor discovery is one of the most important phases of OSPF. Without successfully establishing neighbor relationships, routers cannot exchange routing information. The successful output confirmed that all routers recognized their adjacent routers and were ready to exchange network topology information.

### Solution

The routers successfully established OSPF neighbor relationships, confirming that the routing protocol was operating correctly.

# Step 4: Dynamic Route Learning

Once OSPF neighbors were established, routers exchanged Link-State Advertisements (LSAs), allowing each router to learn routes to remote networks automatically. The updated routing tables were verified using:

```bash
show ip route
```

### Practical Analysis

Unlike static routing, no manual route configuration was required. OSPF automatically calculated the best available paths and updated the routing tables whenever new routing information was received. This demonstrates the self-learning capability of dynamic routing protocols and their ability to simplify network administration.

### Solution

The routing tables successfully contained dynamically learned routes from neighboring routers, proving that OSPF was functioning correctly and enabling end-to-end connectivity.

# Final Result

The experiment successfully demonstrated the implementation of dynamic routing using OSPF over Docker-based FRRouting routers. The virtual routers automatically established neighbor relationships, exchanged routing information, and maintained updated routing tables without requiring manual route configuration.

# Conclusion

This experiment successfully replaced static routing with dynamic routing using OSPF, improving network scalability, adaptability, and ease of management. The Docker-based implementation provides a realistic virtual networking environment while establishing the routing infrastructure required for subsequent experiments involving Policy-Based Networking, ANIMA, Autonomic Service Agents (ASA), and the proposed **Cognitive Autonomic Service Agent (C-ASA)**.
